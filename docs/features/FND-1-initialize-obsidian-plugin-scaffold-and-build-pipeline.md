# FND-1: Initialize Obsidian plugin scaffold and build pipeline

**Story**: Establish the initial Obsidian plugin repository scaffold so contributors can build, lint, typecheck, and test from a single consistent workflow.
**Epic**: Epic 1 — Plugin Foundation and Runtime Shell
**Size**: Small
**Status**: Open

---

## 1. Summary

This story creates the baseline project structure and tooling required for all plugin development work. It defines the Obsidian plugin entrypoint shape, metadata files, and command-line scripts so the codebase can reliably produce `main.js` and satisfy development quality checks.

FND-1 is the first dependency in the backlog execution order. Stories in Epic 1 and later epics assume a stable scaffold already exists, including manifest/version metadata and working build/lint/test/typecheck commands. Completing this story removes setup ambiguity and lets follow-on stories focus on runtime behavior instead of project bootstrapping.

The design principle is minimal-but-production-aligned scaffolding: include only what is necessary to run and verify the plugin shell, while deferring feature logic (views, commands, indexing, providers, chat orchestration) to subsequent stories. This keeps story boundaries clear and avoids early over-engineering.

## 3. Frontend Flow

### 3a. Component / Data Hierarchy

```
Obsidian App
└── Community Plugin Loader
    ├── manifest.json
    ├── versions.json
    └── main.js (generated by esbuild from src/main.ts)
        └── ObsidianAIPlugin (minimal lifecycle shell)
```

### 3b. Props & Contracts

| Component / Hook | Props / Signature | State | Notes |
|------------------|-------------------|-------|-------|
| `ObsidianAIPlugin` | `class ObsidianAIPlugin extends Plugin` | Lifecycle state managed by Obsidian | Minimal `onload()` and `onunload()` shell only |
| Build pipeline | `npm run build` | Build pass/fail | Produces root `main.js` from TypeScript entrypoint |
| Dev/watch pipeline | `npm run dev` | Watch pass/fail | Rebuilds on source changes for rapid local iteration |
| Lint/typecheck/test pipelines | `npm run lint`, `npm run typecheck`, `npm run test` | Command pass/fail | Baseline quality gates that must run before feature work |

### 3c. States (Loading / Error / Empty / Success)

| State   | UI Behavior |
|---------|-------------|
| Loading | Build/lint/test/typecheck command is running; no in-plugin UI flow is in scope yet |
| Error   | Command exits non-zero with actionable output (compile, lint, or test failure) |
| Empty   | Plugin has no feature panes/commands yet beyond scaffold shell; expected for this story |
| Success | `main.js` is generated and plugin shell loads without startup crash in Obsidian |

No end-user frontend feature flow is implemented in FND-1; this section describes scaffold runtime behavior only.

## 5. Acceptance Criteria Checklist

### Phase A: Scaffold and Metadata Foundation

- [ ] **A1** — Obsidian metadata files are valid and aligned
  - `manifest.json` and `versions.json` exist with consistent plugin version values.
  - `manifest.json` includes a valid `minAppVersion` compatible with project requirements.

- [ ] **A2** — TypeScript plugin entrypoint shell exists
  - `src/main.ts` defines a default plugin class extending Obsidian `Plugin`.
  - `onload()` and `onunload()` are present as minimal lifecycle methods with no heavy runtime logic.

- [ ] **A3** — Baseline repository config is in place
  - `package.json`, `tsconfig.json`, and `.gitignore` are present and internally consistent.
  - `npm install` completes without manual patch steps.

### Phase B: Build and Watch Pipeline

- [ ] **B1** — Build command emits Obsidian-compatible artifact
  - `esbuild.config.mjs` compiles `src/main.ts` to root-level `main.js`.
  - `npm run build` exits successfully and produces fresh output.

- [ ] **B2** — Watch mode supports rapid iteration
  - `npm run dev` runs in watch mode without immediate runtime/config failures.
  - Editing `src/main.ts` triggers rebuild output.

- [ ] **B3** — Artifact can be loaded by Obsidian plugin loader
  - `main.js`, `manifest.json`, and `versions.json` can be placed in a vault plugin folder.
  - Enabling the plugin in Obsidian does not produce startup load errors.

### Phase C: Quality Tooling Baseline

- [ ] **C1** — Lint workflow is configured and runnable
  - ESLint config files exist and target TypeScript source.
  - `npm run lint` exits successfully on scaffold code.

- [ ] **C2** — Typecheck workflow validates scaffold
  - `npm run typecheck` runs `tsc --noEmit` (or equivalent) and exits successfully.
  - Obsidian typing references resolve without local hacks.

- [ ] **C3** — Test workflow executes baseline test
  - `vitest.config.ts` is present and wired to npm test script.
  - `npm run test` executes at least one smoke test and exits successfully.

### Phase Z: Quality Gates

- [ ] **Z1** — `npm run build` passes with zero TypeScript errors in all workspaces
- [ ] **Z2** — `npm run lint` passes (or only has pre-existing warnings)
- [ ] **Z3** — No `any` types in any new or modified file
- [ ] **Z4** — All client imports from shared use `@shared/types` alias (not relative paths)

## 6. Risks & Tradeoffs

| # | Risk / Tradeoff | Mitigation |
|---|-----------------|------------|
| 1 | Over-scaffolding can lock in early structure before runtime needs are proven | Keep scope minimal and postpone non-essential abstractions to later stories |
| 2 | Misconfigured build output can create artifacts Obsidian cannot load | Validate output path/format and perform vault-level plugin load check |
| 3 | Toolchain drift between scripts and configs can block contributors | Keep scripts/configs explicit and verify all commands in one pass before completion |
| 4 | `Z4` quality gate may be not-applicable in this plugin-only stage | Keep gate for template consistency and treat applicability explicitly during implementation verification |

## Implementation Order

1. `package.json`, `tsconfig.json`, `.gitignore` — establish project scripts and base TypeScript configuration (covers A3).
2. `manifest.json`, `versions.json` — add Obsidian plugin metadata and compatibility map (covers A1).
3. `src/main.ts` — implement minimal plugin lifecycle shell class (covers A2).
4. `esbuild.config.mjs` — wire build and watch behavior to generate `main.js` (covers B1, B2).
5. `.eslintrc.cjs`, `.eslintignore` — configure baseline linting workflow (covers C1).
6. `vitest.config.ts`, `src/__tests__/smoke.test.ts` — wire baseline test harness and smoke test (covers C3).
7. **Verify** — run `npm install`, `npm run build`, `npm run typecheck`, `npm run lint`, and `npm run test` (covers B1, C1, C2, C3, Z1, Z2, Z3).
8. **Verify in Obsidian** — load generated artifact set in a test vault and enable plugin (covers B3).
9. **Final verify** — confirm no out-of-scope edits and all acceptance criteria are demonstrably met (covers Phase Z).
